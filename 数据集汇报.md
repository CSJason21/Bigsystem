# 数据集汇报

## 1. 任务背景

根据任务日志，我当前负责的是“算力网络协同预测与分配”部分，页面重点主要有两块：

1. 多维度算力资源服务需求预测
2. 联邦智能调度资源分配结果展示

因此，评估数据集时不能只看“是否为集群数据”，而要看它是否支持以下关键内容：

- 任务需求侧：CPU、GPU、内存、任务开始/结束、等待时长
- 节点资源侧：节点资源容量、节点负载、资源占用情况
- 调度结果侧：任务被分配到哪个节点、分配了多少资源、调度后的效果
- 预测展示侧：是否具备可做时间序列预测的历史数据
- 扩展维度：带宽、存储、节点间流向、调用关系

## 2. 两个仓库的总体结论

### 2.1 `cluster-data-master`

这是 Google Cluster Trace。

特点：

- 偏调度事件、任务实例、CPU/内存资源请求与使用
- 本地仓库中主要是 README、proto、说明文档，没有实际原始数据文件
- 2019 版本数据量很大，官方说明主要通过 BigQuery 使用
- 缺少 GPU 维度，也没有直接对应“源/目的节点流向”“带宽分配”“存储分配”的字段

结论：

- 不适合作为我这部分的主数据源
- 最多可作为调度事件建模、CPU/内存预测思路的参考数据

### 2.2 `clusterdata-master`

这是 Alibaba Cluster Trace。

特点：

- 与“算力网络协同预测与分配”更接近
- 包含 GPU 调度、机器规格、任务请求、任务调度结果等更贴近场景的数据
- 但不同子数据集覆盖的维度不同，需要组合使用

结论：

- 可以作为我这部分的主要数据来源
- 但不能指望单一子数据集一次性覆盖全部页面字段

## 3. 各子数据集适配性分析

### 3.1 `cluster-trace-gpu-v2023`

本地已有可直接使用的 CSV：

- `openb_node_list_gpu_node.csv`
- `openb_pod_list_default.csv`

可以提供：

- 节点资源池：CPU、内存、GPU 数量、GPU 型号
- 任务请求：CPU、内存、GPU 数量、GPU share、QoS
- 调度时间信息：`creation_time`、`scheduled_time`、`deletion_time`

适合支持：

- 资源池展示
- 任务需求展示
- 任务等待时长、运行时长统计
- 资源分配结果页
- 节点负载均衡效果页的基础版本

不足：

- 没有真实带宽/存储时序
- 没有真实节点到节点通信流向
- 更偏调度输入与调度结果，不是完整监控时序集

判断：

- 这是当前最适合先落地的演示数据
- 特别适合优先完成“资源分配结果面板”

### 3.2 `cluster-trace-gpu-v2020`

可提供的核心表结构：

- `pai_job_table`
- `pai_task_table`
- `pai_instance_table`
- `pai_sensor_table`
- `pai_machine_spec`
- `pai_machine_metric`

可以提供：

- 任务请求：`plan_cpu`、`plan_mem`、`plan_gpu`
- 机器能力：CPU、内存、GPU 容量
- 机器运行指标：`machine_cpu`、`machine_gpu`、`machine_net_receive`
- 实例生命周期：`start_time`、`end_time`

适合支持：

- 任务需求字段设计
- 任务与节点的关联
- 调度效果指标设计
- 机器负载与任务资源占用展示

不足：

- 许多指标是实例生命周期平均值，不是连续时间序列
- 没有真实 IP
- 网络字段是聚合指标，不足以还原真实节点流图

判断：

- 适合做“任务需求”和“调度结果”的增强版
- 可作为 `v2023` 的补充数据源
- 不适合单独完成严格意义的历史-当前-未来连续预测曲线

### 3.3 `cluster-trace-v2018`

从 schema 可见，它包含：

- `cpu_util_percent`
- `mem_util_percent`
- `net_in`
- `net_out`
- `disk_io_percent`

优势：

- 可以补带宽和磁盘维度

不足：

- 本地目前只有 schema 和说明，没有原始数据
- 没有 GPU 维度

判断：

- 如果后续页面必须补“带宽/磁盘”指标，可以引入它
- 但它不适合作为我这部分的主数据集

### 3.4 `cluster-trace-microservices-v2021/v2022`

可提供：

- 节点与微服务 CPU/内存利用率时序
- 调用速率（MCR）
- 响应时间（RT）
- 调用图（Call Graph）

优势：

- 更适合做流量传播、调用关系、服务链路可视化

不足：

- 不是 GPU 任务调度数据
- 不能单独承担“资源分配结果页”

判断：

- 如果后续要补“通信关系”“服务流量”“调用链路”，它很有价值
- 但它只能作为补充数据集

## 4. 能否满足当前任务

### 4.1 如果目标是先完成前端演示版

可以满足。

推荐主方案：

- 主数据集：`cluster-trace-gpu-v2023`
- 补充数据集：`cluster-trace-gpu-v2020`
- 可选补充：`cluster-trace-v2018`、`microservices-v2021/v2022`

这套方案可以支持：

- 资源需求展示
- 调度分配结果展示
- 节点资源池展示
- 等待时长 / 运行时长 / 负载效果展示

### 4.2 如果目标是单一数据集真实覆盖全部字段

不能满足。

因为我当前想展示的字段包含：

- GPU 需求与使用
- 带宽流量
- 存储配额或磁盘使用
- 源节点/目的节点流向
- 历史/当前/未来预测曲线
- 调度结果与负载均衡效果

当前这两个仓库中，没有任何一个单独数据集能完整覆盖全部维度。

## 5. 我的部分应如何确定数据维度

数据维度不应直接跟着现有字段走，而应先按页面交付目标定义“最小可交付数据结构”。

### 5.1 需求预测面板的数据维度

建议固定为：

- 时间维度：`hour / day / week`
- 实体维度：`cluster / node / pod(task) / gpu_type / qos`
- 核心指标：
  - `gpu_demand`
  - `cpu_demand`
  - `memory_demand`
- 可选补充：
  - `bandwidth_mbps`
  - `wait_time_sec`
  - `running_task_count`
- 预测对比：
  - `actual`
  - `predicted`

建议前端统一结构：

```ts
type DemandPoint = {
  bucket: string
  granularity: 'hour' | 'day' | 'week'
  gpuDemand: number
  cpuDemand: number
  memoryDemand: number
  bandwidthMbps?: number
  runningTasks?: number
  predictedGpuDemand?: number
  predictedBandwidthMbps?: number
}
```

### 5.2 资源分配结果面板的数据维度

建议固定为：

- 源节点维度：`source_node_id`
- 目标维度：`target_node_id` 或 `task_id`
- 任务维度：`job_id / task_id / qos / gpu_type`
- 分配指标：
  - `allocated_cpu`
  - `allocated_gpu`
  - `allocated_memory`
- 可选补充：
  - `allocated_bandwidth`
  - `allocated_storage`
- 调度效果：
  - `wait_time_sec`
  - `estimated_finish_time`
  - `load_balance_score`

建议前端统一结构：

```ts
type AllocationRecord = {
  sourceNodeId: string
  targetId: string
  jobId: string
  taskId: string
  qos?: string
  gpuType?: string
  allocatedCpu: number
  allocatedGpu: number
  allocatedMemory: number
  allocatedBandwidthMbps?: number
  allocatedStorageGb?: number
  waitTimeSec?: number
  estimatedFinishTimeSec?: number
  loadBalanceScore?: number
}
```

### 5.3 关于“源 IP / 目的 IP”

当前不建议直接使用“源 IP / 目的 IP”作为核心字段，原因如下：

- 公开集群数据通常会脱敏
- 当前这些数据更稳定的标识是 `machine / node / pod / job`
- 所以前端和汇报中更适合写“源节点 / 目标节点”

如果 PPT 必须体现 IP 形式，可在展示层将节点 ID 映射为伪 IP，例如 `10.0.0.x`，并明确说明这是脱敏映射。

## 6. 在没有后端的情况下，如何设计界面并使用该数据集

建议采用“先定数据契约，再接后端”的方式推进。

### 6.1 开发顺序

1. 先定义前端统一数据结构
2. 离线处理 CSV，生成前端直接消费的 JSON
3. 页面先读取本地 JSON / mock 数据
4. 后续再将数据来源替换成 FastAPI 接口

### 6.2 建议先准备的前端数据文件

- `demand-forecast.json`
- `allocation-records.json`
- `node-summary.json`

### 6.3 界面结构建议

#### 顶部筛选区

- 时间粒度：小时 / 天 / 周
- QoS
- GPU 类型
- 任务类型
- 节点范围

#### 中部左侧

- 历史 / 当前 / 预测双轴折线图
- 一条竖线表示当前时刻

#### 中部右侧

- 当前峰值需求
- 未来峰值预测
- 平均等待时长
- 调度建议卡片

#### 下部左侧

- 节点资源池到任务请求的流向图
- 可用 Sankey 或关系图展示

#### 下部右侧

- Top5 高负载节点
- Top5 高需求任务
- 资源分配明细表

这样做的好处是：

- 没有后端时也能完成页面演示
- 后续只需替换数据获取层，不需要推翻页面结构

## 7. 当前最可行的落地路线

### 第一阶段

先用 `cluster-trace-gpu-v2023` 完成：

- 节点资源池展示
- 任务需求展示
- 资源分配结果面板
- 等待时长 / 运行时长基础统计

### 第二阶段

再用 `cluster-trace-gpu-v2020` 补充：

- `plan_cpu / plan_mem / plan_gpu`
- `machine_cpu / machine_gpu / machine_net_receive`
- job / task / instance / machine 关联关系

完成：

- 需求预测的指标设计
- 调度效果分析
- 节点负载与资源利用增强展示

### 第三阶段

按缺口补充数据：

- 缺带宽/磁盘：补 `cluster-trace-v2018`
- 缺调用关系/流量传播：补 `microservices-v2021/v2022`

## 8. 如果当前数据仍不够，后续该如何找数据

后续不应泛找“大型公开数据集”，而应按缺失维度定向寻找：

- 缺 `bandwidth / storage` 时序：找 VM/节点级 CPU、内存、带宽、存储轨迹
- 缺 `task -> node` 调度结果：找带创建/调度/结束时间与 placement 的调度轨迹
- 缺 `service call graph`：找带调用图、调用率、响应时间的数据

## 9. 最终结论

1. `cluster-data-master` 不适合作为我这部分主数据源。
2. `clusterdata-master` 可以支撑我先做演示版，但必须按子数据集组合使用。
3. 当前最稳妥的路线是先用 `cluster-trace-gpu-v2023` 做“资源分配结果页”，再用 `cluster-trace-gpu-v2020` 补“需求预测”和“机器负载”。
4. 页面字段命名应优先使用“节点 / 任务”，不应强行要求真实 IP。
5. 在没有后端前，先做静态数据契约 + 本地 JSON/mock，是当前风险最低、推进效率最高的做法。
