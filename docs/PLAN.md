# 调度中枢拓扑与任务流路演方案

## Summary

这版按“演示系统但逻辑自洽”来做：不追求真实还原中国移动完整算网，而是建立一套稳定、可解释、可从数据库渲染的 4+N+31+X 示意拓扑。

核心原则：

- 数据库保存“结构、层级、坐标、状态”，前端只负责按规则渲染。
- 拓扑只展示当前视角必要节点，避免一个大图塞满所有层级。
- 层级切换不是重新生成假拓扑，而是从同一套全量拓扑中按 `view_id` 取子图。
- 任务分配、算法策略选择用“实时模拟事件流”展示，但事件和结果都来自数据库接口，不走前端写死。
- 页面排版继续保持现有 Ant Design 风格，只优化空间比例和信息层次。

## 拓扑建模方案

采用“一个最大拓扑 + 多个视图子图”的方案。

全量拓扑层级固定为：

```text
全国算力网络
  -> 四大枢纽区域
      -> 省级超算中心
          -> DC 算力节点
              -> 边缘算力节点 / 联邦客户端节点
```

命名统一采用“展示名 + 稳定编码”双层：

- 数据库主键用稳定英文 ID：`hub-jingjinji`、`sc-beijing-01`、`dc-beijing-01`、`edge-shenzhen-01`
- 前端展示名用中文：`京津冀枢纽`、`北京省级超算中心`、`北京 DC-01`、`深圳边缘节点-01`
- 详情面板同时显示中文名称和编码，方便四人页面对齐

建议类型约定：

```text
country        全国算力网络
hub            四大枢纽区域
supercenter    省级超算中心
dc             DC 算力节点
edge           边缘算力节点
client         联邦训练客户端，可与 edge 共用节点，只在训练监控页强调
service        调度器、预测器、策略代理等逻辑组件
```

数据库使用：

- `dim_topology_vertex` 保存所有节点
- `dim_topology_edge` 保存父子关系和链路关系
- `dim_topology_view` 保存视图：`global`、`hub_jingjinji`、`province_beijing`、`task_task-fedtrain-99943`
- `dim_topology_view_vertex` 保存某个视图展示哪些点
- `dim_topology_layout` 保存每个视图下每个点的坐标
- `fact_topology_runtime_state` 保存负载、预测负载、风险等级、是否任务目标

这样可以精准控制结构渲染，不依赖 G6 自动布局。前端禁用 force 布局，优先使用数据库坐标。

## 拓扑展示方案

不要在一个画布里展示所有层级。每个视图只展示 2 到 3 层，保证专业、清楚、少留白。

视图规则：

- 全国视图：全国算力网络 + 四大枢纽 + 每个枢纽下 2 到 3 个代表省级超算中心
- 枢纽视图：某个枢纽 + 该区域省级超算中心 + 重点 DC
- 省级视图：省级超算中心 + DC 节点 + 边缘节点
- 任务视图：只展示当前任务链路：调度中心 -> 枢纽 -> 省级超算 -> DC/边缘目标节点

拓扑视觉映射：

```text
节点大小：层级越高越大
节点颜色：当前负载深浅
外环：预测负载
边框：风险等级
高亮描边：当前任务目标节点
蓝色实线：当前调度链路
橙色虚线：预测重定向或备选链路
灰色细线：普通归属关系
```

拓扑空间优化：

- 拓扑卡片改为“左图右详情”的工作台，高度控制在 560 到 640px
- 右侧详情面板固定宽度 320 到 360px
- 图内减少说明文字，把图例放到卡片标题右侧
- 空白区域用层级摘要小面板补足：当前视图节点数、平均负载、最高风险节点、目标任务节点
- 底部预测曲线不要占一整条长横幅，改成与策略/安全模块组成两列或三列卡片区

## 任务流与算法策略展示方案

因为没有真实任务流，采用“数据库驱动的实时模拟”。

核心思路：

- 数据库预置主线任务 `task-fedtrain-99943`
- 后端接口返回当前任务阶段、分配结果、候选节点评分、聚合策略选择、调度日志
- 前端用定时轮询或虚拟时间推进展示“实时变化”
- 所有变化来自接口返回，不在前端临时编故事

新增或补齐接口：

```text
GET /api/allocation/demo/current-task
GET /api/allocation/demo/task-flow?task_id=task-fedtrain-99943
GET /api/allocation/demo/candidate-scores?task_id=task-fedtrain-99943
GET /api/allocation/strategy/aggregation?task_id=task-fedtrain-99943
```

页面展示方式：

- 拓扑右侧显示“当前任务流”：已接收、资源感知、候选评分、绑定节点、下发执行
- 候选节点评分用小表格展示：资源匹配、负载均衡、时延、风险扣分、总分
- 算法策略模块展示：安全态势提示算法风险，RL 选择 Bulyan，四算法曲线作为结果对比
- 日志窗口只展示关键事件，不刷太多随机文本

推荐主线数据：

```text
任务：task-fedtrain-99943 / 联邦训练-图神经网络
来源：反欺诈中心
数据集：用户行为日志 + 交易记录
目标节点：edge-shenzhen-01
候选节点：edge-shenzhen-01、edge-guangzhou-01、dc-guangdong-01
聚合算法：Bulyan
攻击风险：35% 梯度异常
状态：running，进度 42%
```

任务分配逻辑在演示中解释为：

```text
总分 = 资源匹配 + 负载均衡 + 网络时延 + 可信度风险扣分
```

不需要真调度算法，但数据库要保存评分过程，避免页面显得“凭空决定”。

## 页面编排方案

保持当前模块数量，但重新组织空间：

1. 顶部控制栏  
   保留现有层级视角、预测窗口、数据模式、虚拟时间。

2. 第一屏工作台  
   左侧 65%：拓扑图  
   右侧 35%：当前任务流 + 节点详情/调度日志  
   这块是页面中心。

3. 拓扑下方概览卡  
   三个紧凑卡片：核心 KPI、负载 TOP5、任务类型占比。  
   不要让三个图表抢拓扑主视觉。

4. 决策区  
   左侧：全局安全态势  
   右侧：算法策略决策中心  
   解释“为什么 Bulyan”。

5. 预测与分配区  
   左侧：短时资源预测曲线  
   右侧：候选节点评分 / 分配结果  
   底部曲线不再横跨全屏。

这样路演时顺序是：

```text
先看拓扑全局态势
再看当前任务如何流转
再解释为什么选这个节点
再解释为什么切 Bulyan
最后看预测作为调度依据
```

## 数据落地步骤

1. 整理本地 PostgreSQL 种子数据  
   先建立稳定的 4 层拓扑，不追求真实 31 省全量，可先做 4 枢纽 + 8 省级超算 + 12 DC/边缘节点。

2. 写入视图布局  
   每个 `view_id` 手工配置坐标，保证层级不乱。

3. 改造拓扑接口  
   后端直接返回前端需要的 `nodes`、`edges`、`summary`、`events`。

4. 改造前端拓扑组件  
   以数据库坐标为准，禁用自动层级布局；只保留缩放/自适应，不让 G6 重新排层级。

5. 接入任务流模拟接口  
   页面通过接口拿主线任务、任务阶段、候选评分、策略决策。

6. 新增安全态势和算法策略模块  
   不改变整体风格，只做紧凑卡片和清晰图表。

## Test Plan

- 拓扑全国视图、枢纽视图、省级视图层级不交叉、不混乱。
- 每个视图的节点都来自数据库，不依赖前端 `NODE_META`。
- 修改数据库某节点负载后，前端颜色深浅随接口变化。
- 修改 `task_target_flag` 后，目标节点高亮变化。
- 主线任务能完整展示：接收、评分、分配、下发、运行。
- 算法策略模块能展示 Bulyan 被选中，并和安全态势形成解释闭环。
- 页面第一屏没有大面积无意义留白。
- 预测曲线不再作为全屏长条，而是服务于调度解释。

## Assumptions

- 本期不追求真实中国移动算网完整数据，只做结构可信、命名统一、可路演的示意数据。
- “31” 可以在摘要数字中体现，画布中只展示代表节点，避免视觉拥挤。
- 边缘节点在调度页叫 `edge-*`，在训练监控页可作为联邦客户端解释，不额外造一套客户端命名。
- 算法策略选择、任务分配都是数据库驱动的模拟流程，不做真实算法执行。
- 如果统一数据库第一、第二阶段表足够表达，就不新增表；新增仅限安全态势快照和聚合决策记录。
