# 算力网络系统架构改进方案

> **背景**：组会后师兄提出的系列问题，核心诉求是——系统要能讲一个完整的业务故事，从任务录入到任务结束，链路闭环，每个环节逻辑通顺、细节可查，答辩时问到任何一环都能回答"凭什么、怎么做、结果是什么"。
>
> **改进原则**：最基本的逻辑通顺，细节可查，不要求多真实，数据基本模拟。但功能逻辑、链路闭环、以及答辩时能回答具体细节（根据什么调度、算法怎么决策等）。

## 二、调度算法改进

### 2.1 当前问题

调度只按 CPU 使用率排序，依据太单薄。答辩时无法回答"凭什么分到这个节点"。

### 2.2 改进方案：综合评分公式

每个候选节点算一个综合评分，谁分高谁中标：

```
节点综合评分 = 0.4 × 负载得分 + 0.3 × 匹配得分 + 0.3 × 可信得分

负载得分 = 100 - 当前CPU使用率             （越空闲越高）
匹配得分 = min(节点可用资源 / 任务需求, 1) × 100  （越匹配越高）
可信得分 = 节点可信度评分                   （来自安全层）
```

### 2.3 评分明细表（答辩展示用）

| 候选节点 | CPU使用率 | 负载得分 | 资源匹配 | 匹配得分 | 可信度 | 综合评分 | 决策 |
|----------|----------|---------|---------|---------|--------|---------|------|
| edge-shenzhen-01 | 28% | 72 | 95% | 95 | 98 | **86.7** | ★选中 |
| edge-shanghai-01 | 45% | 55 | 90% | 90 | 92 | 76.6 | |
| edge-xian-01 | 20% | 80 | 88% | 88 | 65 | 77.0 | ⚠可信低 |

**结论**：虽然 xian 更空闲，但可信度太低，综合评分被拉下来。

### 2.4 可视化要求

在调度中枢页面，任务提交后展示**调度决策详情面板**，包含：
- 任务需求（CPU/GPU/内存）
- 候选节点评分表（所有候选节点的各项得分）
- 决策依据说明

### 2.5 两个调度要区分清楚

| 调度类型 | 决策问题 | 算法 | 当前状态 |
|----------|---------|------|---------|
| **资源调度** | 任务分到哪个节点？ | 多目标加权评分 | **需实现** |
| **聚合算法调度** | 联邦训练用什么算法合并梯度？ | RL策略（阈值决策） | 已有 |

两者是串行关系：先决定"去哪跑"（资源调度），再决定"怎么训"（聚合算法）。

## 三、任务需求管理改进

### 3.1 当前问题

任务需求只有一个录入入口，没有任务统计和生命周期追踪功能。和调度中心缺乏联动。

### 3.2 改进方案：拆成两个功能

#### 功能 A：任务录入（输入端）

职责：把业务需求标准化成系统可执行的参数。

| 录入方式 | 业务场景 | 实现 |
|----------|---------|------|
| **手动录入** | 运营人员收到单个业务方需求 | 已有，保留 |
| **批量录入** | 周期性任务批量导入（如每月固定的反欺诈训练） | 新增：上传JSON/CSV，批量创建 |

#### 功能 B：任务统计与生命周期追踪（贯穿全程）

职责：记录并展示每个任务从诞生到结束的完整轨迹。

```
任务生命周期详情（每个任务都要有这条记录）：

录入任务（陈舒婷）
  ├─ 任务ID: task-fedtrain-99943
  ├─ 录入参数: 数据集、α=0.5、ε=2.0、优先级
  ├─ 录入时间: 14:20:00
  │
  ├─ [14:20:05] 提交至调度中枢
  │    └─ 触发原因: 运营人员手动提交
  │
  ├─ [14:20:10] 调度中枢收到，开始评估
  │    ├─ 安全评估结果: B+（数据87/算法82/网络79/系统76）
  │    ├─ 资源感知结果: 候选节点5个
  │    ├─ 算法决策: Bulyan（因恶意比例35%）
  │    └─ 调度决策: 分配到 edge-shenzhen-01（综合评分86.7）
  │
  ├─ [14:20:15] 任务下发执行
  │    ├─ 分配资源: CPU 8核/GPU 1张/内存 32GB
  │    └─ 状态: running
  │
  ├─ [14:25:00] 训练监控反馈
  │    └─ 检测到 edge-xian-01 梯度异常 → 触发算法切换记录
  │
  └─ [16:20:00] 任务完成
       ├─ 最终精度: 89%
       ├─ 训练轮次: 50/50
       └─ 状态: completed
```

### 3.3 和调度中心的联动方式

不是页面跳转，而是**数据流转 + 操作触发**：

```
任务统计页                    调度中枢页
    │                            │
    │  ① 任务列表里点"提交调度"  │
    │ ─────────────────────────→ │
    │                            │  ② 调度中枢收到任务
    │                            │     做决策（安全+资源+算法）
    │                            │
    │  ③ 任务状态更新为"已调度"   │
    │ ←───────────────────────── │
    │  生命周期里新增一条记录     │
    │                            │
    │  ④ 点击任务可跳转到调度中枢 │
    │     看该任务的调度决策详情  │
    │ ─────────────────────────→ │
```

## 四、资源感知和需求预测融入工作流

### 4.1 当前问题

资源感知和需求预测是独立展示的模块，没有作为调度算法的输入，和任务流没有实际联系。

### 4.2 模块定位

| 模块 | 定位 | 对工作流的作用 |
|------|------|---------------|
| **资源感知** | 调度的**眼睛**——看当前 | 提供各节点实时负载，作为调度评分的输入 |
| **需求预测** | 调度的**望远镜**——看未来 | 预测未来负载趋势，影响调度的前瞻性决策 |

### 4.3 融入工作流的三种方式

#### 方式 1：作为调度算法的数据输入（后端联动）

```
任务提交到调度中枢
      ↓
调度算法计算节点评分时：
  ├── 负载得分 ← 来自【资源感知】的实时CPU数据
  └── 预测负载 ← 来自【需求预测】的未来趋势数据
      ↓
输出：选中节点 + 评分明细（里面引用了资源和预测的数据）
```

#### 方式 2：在调度决策详情里引用（可视化联动）

在调度决策详情面板里，明确标注数据来源：

```
edge-shenzhen-01 综合评分 86.7
  ├── 负载得分 72  ← 数据来源：资源感知（当前CPU 28%）
  ├── 预测负载     ← 数据来源：需求预测（1小时后预计涨到55%，可承接）
  ├── 匹配得分 95  ← 资源充足
  └── 可信得分 98  ← 网络安全层
```

#### 方式 3：预测预警触发调度调整（主动联动）

```
需求预测发现：未来2小时全网算力需求涨40%
      ↓
触发预警：当前空闲资源不足
      ↓
系统建议：将低优先级任务迁移到边缘节点 / 建议扩容
      ↓
在调度中枢弹出预警提示
```

### 4.4 可视化方式

不是跳转，是"数据引用 + 下钻"：

```
调度中枢页面
  ├── 调度决策详情（引用了资源感知和预测的数据）
  │     ↓ 点击"查看节点详情"
  │     └── 跳转到资源感知页，看该节点的实时监控
  │
  └── 预测预警提示（来自需求预测）
        ↓ 点击"查看预测详情"
        └── 跳转到需求预测页，看趋势曲线
```

## 五、算法决策的架构定位

### 5.1 结论：算法决策引擎独立，但展示在调度中枢

| 层面 | 归属 | 说明 |
|------|------|------|
| **算法逻辑（后端）** | **独立服务** | `algorithm_decision_engine.py` 已经是独立服务 |
| **算法展示（前端）** | **在调度中枢页面** | 因为算法决策是调度的核心环节 |

### 5.2 为什么算法不能完全独立成一个页面

因为算法决策和资源调度是一条连续的决策链：

```
任务提交 → ①评估安全 → ②选算法 → ③选节点 → 下发执行
                          ↑          ↑
                     这两步是连贯的，拆开会打断决策流程
```

### 5.3 算法决策中心在调度中枢里的位置

```
调度中枢页面
  ├── 区块1：全局态势感知（安全雷达 + 资源负载 + 预测）
  ├── 区块2：算法决策中心  ← 独立区块，有自己的完整逻辑
  │    ├── 当前算法：Bulyan（RL自动选）
  │    ├── 决策依据：恶意比例35% + 安全评分82
  │    ├── 4算法收敛曲线对比
  │    └── 决策日志
  ├── 区块3：调度决策详情（节点评分 + 分配结果）
  └── 区块4：拓扑图（全网视图）
```

### 5.4 四个关键问题的回答

**Q1：影响算法的是什么？**

```
影响算法决策的因素（3个输入）：
  ├── 恶意节点比例（35%）     ← 来自训练监控的检测
  ├── 攻击类型（梯度反转）     ← 来自训练监控的检测
  └── 算法层安全评分（82分）   ← 来自安全态势评估
```

**Q2：为什么需要算法？**

> 联邦学习有多个客户端一起训练，如果其中一部分是恶意的（投毒攻击），不同的聚合算法抵抗能力不同。不选对的算法，全局模型会被污染，精度从89%崩溃到61%。

**Q3：算法是怎样决策的？**

基于恶意比例的阈值决策（模拟 RL 策略）：

```
恶意 ≤10%  → FedAvg（无需防御，省算力）
10%~20%   → GeometricMedian（中等防御）
20%~25%   → Krum（较强防御）
>25%      → Bulyan（最强防御）
安全<70分 → 强制Bulyan（安全联动）
```

**Q4：算法安全是什么？**

算法安全是算法决策的约束条件和反馈指标：
- 安全评分低 → 触发算法升级（安全→决策）
- 切换到更鲁棒算法 → 安全评分提升（决策→安全）

## 六、安全量化评估方案

### 6.1 当前问题

4层安全评分是 hardcode 的常量，无法解释计算过程。答辩时无法回答"87分怎么来的"。

### 6.2 两个层级都要：全局安全 + 任务安全

```
┌─ 全局安全态势 ─────────────────────────────┐
│  （面向运营管理员，看全网健康状况）            │
│                                              │
│  数据安全 87 │ 算法安全 82 │ 网络安全 79     │
│                                              │
│  作用：影响【调度策略的整体方向】              │
│  举例：全局算法安全低 → 所有新任务默认用Bulyan│
└──────────────────────────────────────────────┘
                      ↓
┌─ 任务级安全评估 ────────────────────────────┐
│  （面向单个任务，看这个任务安不安全）          │
│                                              │
│  task-99943 的安全评估：                      │
│  ├── 数据安全：ε=2.0 → 评分85                │
│  ├── 算法安全：用Bulyan + 恶意35% → 评分90   │
│  ├── 网络安全：目标节点可信度98 → 评分95      │
│  └── 综合任务安全分：88（A）                  │
│                                              │
│  作用：影响【这个任务的具体调度决策】          │
│  举例：任务安全分低 → 拒绝调度或要求人工确认  │
└──────────────────────────────────────────────┘
```

### 6.3 两个层级的关系

| 维度 | 全局安全 | 任务安全 |
|------|---------|---------|
| **粒度** | 全网 | 单个任务 |
| **数据来源** | 全网汇总统计 | 该任务的具体参数 |
| **更新频率** | 持续评估（30秒刷新） | 任务提交时计算 |
| **用途** | 宏观预警，影响默认策略 | 微观决策，影响具体任务分配 |

### 6.4 安全量化的计算公式

每层都要有明确的输入 → 公式 → 输出：

| 安全层 | 输入（数据从哪来） | 计算公式 | 输出 |
|--------|-------------------|---------|------|
| **数据安全** | 全网数据集的 ε 值列表 | `100 - avg(ε_i) × 10`（ε越小越安全） | 0-100 |
| **算法安全** | 当前聚合算法 + 恶意节点比例 | 已有代码实现（鲁棒性评分 - 超出容忍扣分） | 0-100 |
| **网络安全** | 全网节点可信度列表 | `avg(trust_score_i)`，可信度<70的节点额外扣分 | 0-100 |
| **系统安全** | 节点在线率 + 任务成功率 | `online_rate × 0.6 + success_rate × 0.4` | 0-100 |
| **综合** | 4层分数 + AHP权重 | `Σ(层分数 × AHP权重)` → 等级映射 | A+/A/B+/B/C/D |

AHP 权重矩阵：

| 安全层 | 权重 | 说明 |
|--------|------|------|
| 数据安全 | 0.30 | 最重要，数据是核心资产 |
| 算法安全 | 0.25 | 影响训练结果可靠性 |
| 网络安全 | 0.25 | 影响节点可信度 |
| 系统安全 | 0.20 | 基础设施稳定性 |

### 6.5 安全和调度、算法的联动（3条线）

| 联动线 | 触发 | 动作 |
|--------|------|------|
| **安全 → 算法** | 算法层安全分 < 70 | 强制切换到 Bulyan |
| **安全 → 调度** | 某节点可信度 < 70 | 调度评分时大幅扣分，优先避开 |
| **算法 → 安全** | 切换到更鲁棒算法 | 算法层安全分提升（反馈循环） |

```
                安全评估
               ╱        ╲
              ╱          ╲
         调度决策  ←──→  算法决策
              ╲          ╱
               ╲        ╱
                执行下发
```

## 七、任务状态机改进

### 7.1 当前问题

状态映射零散，没有明确的状态机定义，状态跳转逻辑不严谨。

### 7.2 完整状态机

```
                    ┌──────────────────────────────────────┐
                    ↓                                      |
  pending ──→ scheduling ──→ scheduled ──→ running ──→ completed
  (待分配)    (调度中)       (已分配)      (运行中)     (已完成)
                 │               │             │
                 ↓               ↓             ↓
              cancelled       cancelled       failed
              (已取消)        (已取消)        (失败)
```

### 7.3 状态转换触发条件

| 转换 | 触发条件 | 动作 |
|------|---------|------|
| pending → scheduling | 运营人员点击"提交至调度中枢" | 记录流转日志 |
| scheduling → scheduled | 调度算法完成，选定节点+算法 | 写入分配记录 |
| scheduled → running | 资源池确认资源就绪 | 开始训练 |
| running → completed | 训练轮次达到目标 | 记录结果 |
| running → failed | 训练异常/超时 | 记录失败原因 |

### 7.4 落地要求

- 后端定义统一的状态枚举和转换校验
- 每次状态变更写入 `task_flow_log`（任务流转日志表）
- 前端任务列表展示状态，可点击查看状态变更历史

## 九、数据层说明

当前系统不要求真实接入，但要求走后端请求，公用 JSON 数据跑通整个系统，方便路演。

所有改进在数据层的要求：
- 后端接口要返回真实计算过程（公式、输入、输出），而不是 hardcode 结果
- 数据本身可以是 mock 的，但计算逻辑要真实
- 每个接口的返回值要能支撑前端展示"决策详情"和"评分明细"
