# 协同预测预分配数据库设计

## 1. 文档目标

本文档面向当前 `src/pages/ComputingSchedule/PredictionAllocation/index.tsx` 页面，以及后续真实后端接入需求，给出一套可落地的数据库设计方案。

目标不是做“前端随机 mock”，而是建设一套：

- 能存拟真数据
- 能接入后端服务
- 能支撑当前前端正常展示
- 后续能平滑替换为真实生产数据
- 能继续扩展调度、预测、拓扑、资源监控能力

本文档重点覆盖：

- 页面功能到数据库实体的映射
- 各类表设计
- 拓扑、节点、算力资源、任务、预测、日志、链路、策略评估等数据
- 后端 API 如何从数据库取数
- 拟真数据如何流入数据库
- 未来真实数据如何并入

## 2. 当前功能界面需要支撑的核心能力

当前协同预测预分配页面包含以下能力：

1. 全局/区域/省级/单节点视角切换
2. 核心 KPI 展示
3. 负载率 Top5
4. 任务分配类型占比
5. 协同分配拓扑图
6. 点击非算力节点后展示调度日志
7. 点击算力节点后展示节点详情和执行中任务
8. 单节点 CPU/内存/GPU/磁盘监控
9. 底部预测曲线展示
10. 固定时间段查询与实时回放

从数据库角度看，这些功能至少需要五大类数据：

- 拓扑与组织结构数据
- 节点静态元数据
- 节点与链路时序监控数据
- 预测结果数据
- 任务与调度过程数据

## 3. 推荐数据库架构

建议采用 PostgreSQL 作为主库，TimescaleDB 作为时序增强。

原因：

- PostgreSQL 适合管理节点、任务、调度、拓扑、策略、配置等关系型数据
- TimescaleDB 适合管理高频监控指标、预测序列、链路序列、日志序列
- 前后端查询模式比较清晰，既有配置型表，也有时间序列表

推荐逻辑分层：

- `dim_*`：维度表，存放静态或低频变化信息
- `fact_*`：事实表，存放任务、日志、状态、评估等业务事实
- `ts_*`：时序表，存放节点指标、链路指标、预测序列等高频数据
- `vw_*`：视图或物化视图，面向接口聚合返回

## 4. 建模原则

### 4.1 一个页面，多层实体

页面里同时有两类节点：

- 物理/逻辑算力节点：DC、边缘节点、算力池节点
- 控制/调度节点：国家级调度中心、枢纽节点、区域中心、省级大脑、预测引擎、策略代理、网络编排

因此数据库里不要只建“算力节点表”，还要有“统一拓扑顶点表”。

### 4.2 当前值与历史值分离

- 当前展示用“最新快照表/视图”
- 历史趋势用时序表

这样前端展示快，后端查询清晰。

### 4.3 拟真数据与真实数据共存

所有关键事实表建议带上数据来源字段：

- `synthetic_realistic`
- `real_imported`
- `replayed_trace`
- `manual_patch`

后期可以逐步替换，不需要推翻表结构。

## 5. 总体实体关系

建议把整个数据库拆成 9 个主题域：

1. 区域与组织域
2. 拓扑域
3. 节点与资源域
4. 监控时序域
5. 预测域
6. 任务与调度域
7. 流量与链路域
8. 策略评估域
9. 数据接入与仿真域

## 7. 区域与组织域

### 7.1 `dim_region`

描述国家、枢纽、区域、省、市等行政或逻辑区域。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `region_id` | varchar PK | 区域唯一标识，如 `china`、`beijing` |
| `region_name` | varchar | 区域名称 |
| `region_level` | varchar | `country` / `hub` / `region` / `province` / `city` |
| `parent_region_id` | varchar FK | 上级区域 |
| `display_order` | int | 展示顺序 |
| `longitude` | numeric | 经度 |
| `latitude` | numeric | 纬度 |
| `is_active` | boolean | 是否启用 |
| `created_at` | timestamptz | 创建时间 |
| `updated_at` | timestamptz | 更新时间 |

用途：

- 视角切换
- 拓扑挂载
- 节点归属
- 区域级聚合

### 7.2 `dim_topology_view`

定义前端可切换的视图，例如全局、区域、某省、单节点。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `view_id` | varchar PK | 如 `global`、`region_beijing` |
| `view_name` | varchar | 展示名称 |
| `view_kind` | varchar | `global` / `region` / `province` / `node` |
| `region_id` | varchar FK | 该视图主区域 |
| `node_id` | varchar nullable | 单节点视角时绑定的节点 |
| `is_default` | boolean | 是否默认 |
| `is_active` | boolean | 是否启用 |

### 7.3 `dim_topology_view_vertex`

定义某个视图中有哪些顶点，用于稳定前端视角。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | bigserial PK | 主键 |
| `view_id` | varchar FK | 视图 ID |
| `vertex_id` | varchar FK | 拓扑顶点 ID |
| `display_order` | int | 展示顺序 |
| `is_visible` | boolean | 是否显示 |

## 8. 拓扑域

### 8.1 `dim_topology_vertex`

统一描述拓扑上的所有节点，包括调度节点、逻辑节点、算力节点。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `vertex_id` | varchar PK | 顶点 ID |
| `vertex_type` | varchar | `control` / `service` / `compute` / `edge` / `pool` |
| `vertex_role` | varchar | 如 `national_dispatch_center`、`policy_agent`、`dc_compute_node` |
| `vertex_name` | varchar | 展示名称 |
| `subtitle` | varchar | 副标题 |
| `region_id` | varchar FK | 所属区域 |
| `compute_node_id` | varchar nullable FK | 如果是算力节点，则关联算力节点表 |
| `service_code` | varchar nullable | 如果是逻辑组件，可填 `predictor`、`allocator` |
| `status` | varchar | `online` / `offline` / `new` / `degraded` |
| `is_physical` | boolean | 是否物理实体 |
| `is_schedulable` | boolean | 是否可承载任务 |
| `created_at` | timestamptz | 创建时间 |
| `updated_at` | timestamptz | 更新时间 |

说明：

- 前端拓扑图上的“国家调度中心、枢纽、区域中心、省级大脑、预测引擎、策略代理、网络编排、DC、边缘节点”都应该来自这里

### 8.2 `dim_topology_edge`

| 字段 | 类型 | 说明 |

| `capacity_qps` | numeric | 理论任务吞吐 |
| `priority` | int | 优先级 |
| `is_active` | boolean | 是否启用 |

### 8.3 `dim_topology_layout`

用于保持前端布局稳定，不依赖每次前端自动排版。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | bigserial PK | 主键 |
| `view_id` | varchar FK | 视图 ID |
| `vertex_id` | varchar FK | 顶点 ID |
| `x` | numeric | 布局横坐标 |
| `y` | numeric | 布局纵坐标 |
| `size_hint` | numeric | 节点推荐大小 |
| `layout_version` | varchar | 布局版本 |

- 后端给前端返回布局坐标后，页面刷新时拓扑不会跳动

## 17. 拟真数据如何工作

### 17.1 拟真数据不是随机数

拟真数据建议遵循以下原则：

1. 有稳定的区域结构
2. 有真实感的节点能力差异
3. 有时间连续性
4. 有资源与任务的因果关系
5. 有异常事件和重定向逻辑

### 17.2 拟真数据生成思路

建议按下面顺序生成并入库：

1. 先灌静态维度
   - 区域
   - 视角
   - 拓扑顶点
   - 拓扑边
   - 布局坐标
   - 节点元数据
   - GPU 设备
   - 资源池

2. 再灌历史基线
   - 节点 CPU/内存/带宽时序
   - GPU 时序
   - 磁盘时序
   - 链路时序

3. 再灌任务数据
   - 任务主表
   - 任务需求
   - 任务候选评分
   - 任务分配结果
   - 任务执行记录

4. 最后灌预测和调度过程
   - 预测运行
   - 预测点
   - 拓扑运行态
   - 调度日志
   - 重定向事件
   - 流量快照

### 17.3 拟真数据推荐场景

至少准备以下 6 类场景：

1. 平稳日常负载
2. 晚高峰推理突发
3. 训练任务集中提交
4. 某边缘节点离线
5. 新节点接入扩容
6. 预测命中导致提前重定向

这样前端页面的：

- 预测环
- 调度日志
- 离线节点
- 新接入节点
- 重定向边
- 执行中任务列表

都能被真实触发，而不是假写死。

## 18. 数据流如何工作

建议的业务数据流如下。

### 18.1 采集流

1. 节点监控采集器采集 CPU、内存、GPU、带宽、磁盘
2. 网络监控采集器采集链路带宽、延迟、丢包、流数
3. 数据写入 `ts_node_resource_metric_1m`、`ts_gpu_resource_metric_1m`、`ts_network_link_metric_1m`
4. 聚合刷新 `vw_node_runtime_snapshot`

### 18.2 预测流

1. 预测服务读取过去一段时间的节点时序
2. 按节点或区域生成未来 10 分钟到数小时的预测
3. 写入 `fact_forecast_run` 和 `ts_forecast_point`
4. 同时写一份简化快照到 `ts_node_short_forecast_snapshot`
5. 拓扑快照表 `fact_topology_runtime_state` 更新预测负载

### 18.3 调度流

1. 任务进入 `fact_task`
2. 任务需求写入 `fact_task_requirement`
3. 调度器读取：
   - 当前资源快照
   - 短时预测快照
   - 网络链路快照
   - 资源池约束
4. 计算候选节点评分，写入 `fact_task_candidate_score`
5. 选定目标节点，写入 `fact_task_assignment`
6. 执行过程中写入 `fact_task_execution`
7. 产生调度日志，写入 `fact_schedule_log`
8. 如果需要迁移，写入 `fact_reroute_event`
9. 生成链路流数据，写入 `fact_traffic_flow_snapshot`

### 18.4 接口流

1. 后端 API 读取视图和快照表
2. 前端拉取：
   - 节点列表
   - 节点详情
   - 历史趋势
   - 拓扑快照
   - 调度日志
   - 预测曲线
3. 页面完成展示

## 22. 结论

这套数据库设计的核心不是“为当前页面临时造字段”，而是搭一个能长期演进的数据底座。

它有三个关键特点：

1. 统一拓扑建模
   - 控制节点和算力节点一起建模，适合当前拓扑展示

2. 关系型 + 时序型分层
   - 静态元数据、任务、调度用关系型
   - 指标、预测、链路、日志用时序或高频事实表

3. 拟真与真实兼容
   - 先用拟真数据灌库让系统跑起来
   - 后续真实数据逐步替换，不需要重构前端页面
