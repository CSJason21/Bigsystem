# 前端原型数据集说明

## 1. 目标

该脚本用于把 `dataset/` 目录下三个来源不同、时间粒度不同的 CSV 数据，整理成前端可直接消费的 3 份 JSON：

- `dataset/nodes_monitor.json`
- `dataset/task_trace.json`
- `dataset/global_kpi.json`

脚本位置：

- `scripts/generate_mock_json_datasets.py`

运行方式：

```powershell
python scripts/generate_mock_json_datasets.py
```

## 2. 输入文件

### 2.1 `server_monitor.csv`

真实单机探针数据，提供 CPU、内存、磁盘、网络吞吐以及多卡 GPU 利用率。

本次处理将其作为种子节点 `Node_1999 / Alpha` 的基础数据来源。

### 2.2 `disaggregated_DLRM_trace.csv`

任务流数据，核心使用字段：

- `instance_sn`
- `cpu_request`
- `scheduled_time`
- `deletion_time`

其中 `scheduled_time` / `deletion_time` 为相对时间戳，需要映射为绝对时间。

### 2.3 `Load_Balancing_Dataset_with_Tasks.csv`

宏观 KPI 数据源。脚本也兼容 `Load_Balancing_Dataset.csv` 文件名。

该文件本身没有时间列，因此脚本会生成统一的 5 秒时间轴，并按顺序循环填充原始记录。

## 3. 时间基准与窗口

- 绝对时间基准 `T_start` 取自 `server_monitor.csv` 第一行时间
- 当前数据中 `T_start = 2026-04-09 00:04:05`
- 输出窗口长度固定为 2 小时
- 统一时间粒度为 5 秒
- 最终时间步数固定为 1440

说明：

`server_monitor.csv` 在原始首 2 小时内存在明显时间空洞，因此脚本做了一个温和的“微调”处理：

- 优先尝试直接截取首个 2 小时窗口
- 如果样本不足 1440 条，则回退为抽取最早的 1440 条真实探针样本
- 再把这些样本重映射到从 `T_start` 开始的连续 5 秒时间轴上

这样可以保证前端展示拿到稳定、连续、长度固定的时序数据。

## 4. 输出文件说明

## 4.1 `nodes_monitor.json`

用途：

- 节点监控时序图
- 节点列表卡片
- 单节点详情抽屉

生成规则：

- `Node_1999 / Alpha`：直接使用种子探针数据
- `Node_0008 / Beta`：低负载克隆
  - `cpu_usage_pct = Alpha * 0.7 + 随机噪声[-3, 3]`
  - `memory_usage_pct = Alpha * 0.7 + 随机噪声[-3, 3]`
- `Node_0042 / Gamma`：高负载克隆
  - `cpu_usage_pct = Alpha * 1.3 + 随机噪声[-3, 3]`
  - `memory_usage_pct = Alpha * 1.3 + 随机噪声[-3, 3]`
  - 超过 100 的值会截断到 100

新增字段：

- `jitterMs`：`[2, 10]` 范围内的随机浮点数
- `packetLoss`：
  - 当 `cpu_usage_pct > 90` 时，在 `[0.5, 2.0]` 之间随机生成
  - 其余时刻为 `0`

主要字段：

- `timestamp`
- `node_id`
- `node_name`
- `cpu_usage_pct`
- `memory_usage_pct`
- `memory_used_mb`
- `disk_usage_pct`
- `net_in_kbps`
- `net_out_kbps`
- `jitterMs`
- `packetLoss`
- `gpu_0_util_pct ~ gpu_3_util_pct`
- `gpu_0_mem_used_mb ~ gpu_3_mem_total_mb`

## 4.2 `task_trace.json`

用途：

- 任务甘特图
- 调度事件流
- 节点任务绑定展示

时间对齐规则：

```text
真实时间 = T_start + (原始时间 - min_time)
```

其中：

- `min_time` 为 `scheduled_time` 的最小值
- `scheduled_at` 为映射后的任务开始时间
- `deletion_at` 为映射后的任务结束时间

过滤规则：

- 仅保留与 2 小时窗口有重叠的任务

补全规则：

- 若 `deletion_time` 缺失，或早于 `scheduled_time`
- 则从已有有效任务时长分布中抽样补齐
- 抽样时长被限制在 300 到 3600 秒之间，避免异常超长或超短

节点绑定规则：

- 新增 `target_node_id`
- CPU 请求越高，越倾向绑定到 `Node_0042 / Gamma`
- 中低负载任务更容易分配给 `Node_1999 / Alpha` 与 `Node_0008 / Beta`

主要字段：

- `task_id`
- `target_node_id`
- `cpu_request`
- `gpu_request`
- `memory_request`
- `scheduled_at`
- `deletion_at`
- `duration_sec`

## 4.3 `global_kpi.json`

用途：

- 全局 KPI 趋势图
- 平台总体吞吐与响应看板

生成规则：

- 使用 5 秒步长生成 2 小时时间轴
- 按顺序读取宏观 CSV
- 若 1440 行不足，则循环拼接原始数据直到填满

覆盖字段：

- `cpu_utilization`
  - 不再使用原 CSV 值
  - 直接取 3 个虚拟节点当时刻 CPU 利用率均值
- `load_variance`
  - 新增字段
  - 取 3 个虚拟节点 CPU 利用率的标准差

主要字段：

- `timestamp`
- `response_time`
- `throughput`
- `cpu_utilization`
- `load_variance`
- `task_count`
- 其余宏观容量 / 需求 / 优先级字段原样保留

## 5. 结果规模

默认输出规模如下：

- `nodes_monitor.json`：`1440 * 3 = 4320` 条
- `task_trace.json`：取决于 2 小时窗口内实际任务数
- `global_kpi.json`：`1440` 条

## 6. 可直接给前端使用的建议

- `nodes_monitor.json` 可直接按 `node_id` 分组画多节点折线图
- `task_trace.json` 可按 `target_node_id` 画泳道或甘特条
- `global_kpi.json` 可直接驱动大盘顶部 KPI 和趋势图

如果后续前端需要更贴近组件的字段名（例如统一改成 camelCase），建议只在脚本里做字段映射，不要再手工改 JSON。
